# 开发路线图

## 1. 总体原则

本项目开发顺序必须遵守以下原则：

- 先完成基础骨架，再进入业务开发
- 先完成单人可用，再进入多人实时能力
- 先完成后端与数据库主链路，再补前端体验
- 文档与开发记录同步维护，不要最后集中补写

---

## 2. 阶段划分

### Phase 1：需求冻结与工程骨架

目标：

- 固定首版范围
- 建立项目目录
- 建立后端基础骨架
- 建立前端基础骨架
- 建立 notes 文档体系

输出物：

- `backend/` 工程
- `frontend/` 工程
- `notes/架构/` 规划文档
- `plans/` 总计划索引

### Phase 2：数据库与认证

目标：

- 设计数据库表结构
- 输出 `schema.sql`
- 输出测试数据 `init.sql`
- 打通注册、登录、退出、当前用户、忘记密码

输出物：

- `sql/schema.sql`
- `sql/init.sql`
- Auth 相关接口
- User 基础接口

### Phase 3：TEAM 与知识库

目标：

- 支持 TEAM 创建、查看、加入、退出
- 支持一个用户加入多个 TEAM
- 支持个人知识库与 TEAM 知识库
- 支持知识库成员授权

输出物：

- TEAM 模块接口
- 知识库模块接口
- TEAM / 知识库基础页面

### Phase 4：文章与编辑

目标：

- 实现文章 CRUD
- 实现图片上传
- 接入 Milkdown 单人编辑
- 完成文章详情页与编辑页

输出物：

- Article 接口
- Upload 接口
- Milkdown 编辑页

### Phase 5：聊天与协同

目标：

- 建立 WebSocket 连接体系
- 完成文档侧边栏聊天
- 接入 Yjs 协同
- 完成在线成员展示与基本保存策略

输出物：

- WebSocket 模块
- 聊天侧边栏
- 协同编辑能力

### Phase 6：收尾与答辩准备

目标：

- 首页与导航收口
- 权限状态与异常页收口
- README 与文档完善
- 开发记录整理
- 演示流程整理

输出物：

- README
- 完整 notes 文档
- 开发记录
- 演示脚本

## 4. 每阶段验证要求

### Phase 1 完成标准

- 工程可以启动
- 文档结构已建立
- 目录结构已稳定

### Phase 2 完成标准

- 数据库脚本可初始化
- 用户能完成注册、登录、退出
- 当前用户接口可用
- 忘记密码链路可演示

### Phase 3 完成标准

- 用户可加入多个 TEAM
- TEAM 列表与详情可展示
- 可创建个人知识库和 TEAM 知识库
- 可配置知识库成员角色

### Phase 4 完成标准

- 可创建和编辑文章
- 可查看文章详情
- Milkdown 单人编辑可稳定保存

### Phase 5 完成标准

- 文档聊天实时可用
- 在线成员可见
- 同一文章可被多人进入协同
- 基础同步可演示

### Phase 6 完成标准

- 页面可完整演示主要业务
- 文档齐全

---

## 7. 文档联动要求

开发过程中必须同步更新：

- `notes/架构/`
- `notes/记录/开发记录.md`
- `README.md`

不能等到项目完成后再集中补写。
